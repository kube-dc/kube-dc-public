# Dedicated GPU VM workflow

Use this workflow only when the user explicitly requests a whole-device
Dedicated GPU VM.

## Preconditions

1. Query `GET /api/capabilities/{project-namespace}/gpus` with the caller's
   Kube-DC bearer token or open the VM wizard.
2. Continue only when `vmCreationEnabled` is `true` and the selected
   Dedicated GPU VM profile has `canCreate: true`.
3. Confirm device quota headroom, CPU, memory, storage, and compatible physical
   capacity. Quota is not a reservation; the VM can wait for a device.
4. Use an operator-qualified guest image and exact signed guest-driver package.

Do not attach a native host-device resource with `kubectl`. Raw Kubernetes
submission bypasses the independent product creation flag and can expose
provider resource names. Use the Kube-DC VM wizard or the backend VM create flow,
which authorizes the caller and converts the stable profile alias to the live
catalog resource only after validating quota, fleet allowlist, and capacity.

## Stable review-manifest delta

The VM review YAML carries only the stable product profile:

```yaml
metadata:
  labels:
    kube-dc.com/gpu-profile: nvidia-v100-passthrough
spec:
  template:
    metadata:
      labels:
        kube-dc.com/gpu-profile: nvidia-v100-passthrough
    spec:
      evictionStrategy: None
      domain:
        devices:
          gpus:
            - name: gpu0
              deviceName: kube-dc.com/nvidia-v100-passthrough
```

Merge that shape into the normal VM manifest generated by the Kube-DC wizard.
Keep device names unique when more than one device is supported (`gpu0`,
`gpu1`, ...). Do not add `nodeName`, `nodeSelector`, node affinity,
generic `hostDevices`, a runtime class, or `LiveMigrate`.

For an API client, submit the complete review YAML to the authenticated
validation route before creating any root volume:

```http
POST /api/create-vm/{project-namespace}/vmi/validate
Authorization: Bearer {kube-dc-jwt}
Content-Type: application/yaml
```

Then use the existing Kube-DC VM create transaction, ending with the same YAML
at `POST /api/create-vm/{project-namespace}/vmi`. The authoritative create
route repeats preflight and canonicalizes the stable alias immediately before
Kubernetes submission. Do not replace this flow with a direct KubeVirt create.

## Lifecycle and verification

- GPU VMs cannot live migrate. Shared RBD storage protects the disk but does not
  make the attached device migratable.
- Stop releases the device after the VMI terminates. Pause and guest reboot keep
  it allocated.
- A later start can wait for capacity and can attach a different physical
  device. Never rely on node, PCI, or GPU identity.
- Verify attachment in the guest before installing a driver. Install only the
  operator-approved proprietary package for the V100 pilot; do not install a
  vGPU/GRID guest package.

After the VM reaches Running:

```bash
kubectl get vm,vmi -n {project-namespace} {vm-name}
```

Inside a qualified Linux guest:

```bash
lspci -nn | grep -Ei 'NVIDIA|3D controller|VGA'
nvidia-smi --query-gpu=name,driver_version,memory.total --format=csv,noheader
```

A successful `nvidia-smi` check proves enumeration, not application
compatibility. Run the operator's pinned CUDA smoke test. During maintenance,
stop the VM, wait for device release, and start it only after compatible
capacity returns.
